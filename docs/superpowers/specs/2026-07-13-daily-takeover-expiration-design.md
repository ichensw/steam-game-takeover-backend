# 每日接龙自动结束设计

## 目标

避免“每日”类型接龙长期保持进行中状态，影响用户和管理员查看。管理员可以配置每日接龙从创建起保持进行中的天数；超过期限后，后端自动把接龙状态更新为“已结束”。

## 已确认语义

- 只处理 `schedule_type = ScheduleDaily` 的接龙。
- 有效期从 `gmt_create` 开始计算，修改接龙不会重新计时。
- 未配置时默认有效期为 10 天。
- 可配置范围为 1 至 365 天，整数。
- 到期条件为 `gmt_create <= now - expirationDays`。
- 只把 `is_deleted = false` 且 `takeover_state = TakeoverStateNormal` 的记录更新为 `TakeoverStateClosed`。
- 不修改 `is_deleted`，不物理删除接龙，不删除成员、活动、总结或其他关联记录。
- 已结束和已删除记录不重复处理。

## 配置

沿用 `ttw_app_config`，新增配置键 `daily_takeover_expiration_days`。后端读取规则：

- 配置不存在、为空、不是整数或超出 1 至 365 时使用默认值 10。
- 管理后台 `GET /api/admin/settings` 返回 `dailyTakeoverExpirationDays`。
- 管理后台 `PUT /api/admin/settings` 接受可选整数 `dailyTakeoverExpirationDays`；超出范围返回 `400 PARAM_INVALID`，不保存该值。
- 设置保存响应回传规范化后的有效值。

前端在现有“系统设置”页面增加数字输入：

- 标签：“每日接龙有效天数”。
- 最小值 1，最大值 365，步长 1。
- 辅助说明：“每日类型接龙从创建时间起超过该天数后自动标记为已结束，默认 10 天。”
- 加载时缺失值显示 10；保存时发送整数。

## 自动结束任务

新增每日接龙过期 worker，沿用现有服务后台任务模式：

- 服务启动后立即执行一次。
- 之后每小时执行一次。
- 每次执行时重新读取配置，因此修改设置后最迟一小时生效；服务重启时立即生效。
- 使用单条批量 `UPDATE`：按每日类型、未删除、进行中和创建时间截止点筛选。
- 更新失败写应用错误日志，不终止服务；下一轮继续重试。
- 成功且更新数量大于 0 时写信息日志，包含更新数量和采用的有效天数，不记录用户数据。
- worker 监听进程 context，服务关闭时停止 ticker。

现有指定日期和日期范围接龙的按请求过期逻辑保持不变。每日接龙使用独立 worker，避免给现有多个请求路径增加配置查询和额外写操作。

## 启动与数据流

`cmd/server/main.go` 在创建 Handler 和 Router 后，与提醒 worker 一样启动每日接龙过期 worker：

1. worker 读取 `daily_takeover_expiration_days`。
2. 计算 `cutoff = now.AddDate(0, 0, -days)`。
3. 批量更新符合条件的 `ttw_takeover`。
4. 记录执行错误或更新数量。
5. 等待下一小时或 context 取消。

部署后首次启动会立即处理已有的、创建超过默认或当前配置天数的每日接龙。这是预期行为，不进行历史数据回填或删除。

## 错误与并发

- SQL 更新带完整状态条件，重复执行是幂等的。
- 多实例同时执行时，首个实例更新状态，后续实例匹配不到进行中记录，不会破坏数据。
- 配置读取失败按现有 `appConfigValue` 行为回退默认 10 天，保证任务仍可运行。
- 设置接口先校验再保存，防止无效配置进入数据库。

## 测试

后端测试覆盖：

- 配置缺失、非法和越界时返回默认 10。
- 有效配置 1 和 365 能正确读取。
- 设置接口拒绝 0、366 和非整数输入。
- 清理 SQL 只匹配每日、未删除、进行中的记录。
- 边界 `gmt_create == cutoff` 会结束，晚于 cutoff 不会结束。
- worker 启动立即执行并能响应 context 取消；周期部分通过可注入执行函数或直接测试单次清理函数，避免依赖真实一小时计时。

前端测试覆盖：

- 设置规范化在字段缺失时使用 10。
- 有效值保存为整数。
- 输入控件限制 1 至 365。

全量验证包括 `go test -count=1 ./...`、`go vet ./...`、`npm test` 和 `npm run build`。浏览器检查系统设置页在桌面和移动视口无重叠、字段文案和默认值正确、控制台无新增错误。

## 发布与回滚

- 不新增数据库表或迁移。
- 部署主后台后端与前端，Nginx 路径不变，预期无需修改配置。
- 替换前保留后端二进制和前端 `dist` 的时间戳备份。
- 发布后验证服务健康、设置接口字段、系统设置页面以及一条测试查询确认已有超期每日接龙已变为已结束。
- 若出现异常，恢复前一版二进制和前端目录并重启服务；已被标记为已结束的数据不会自动恢复，必要时可由管理员按明确 ID 手工恢复，禁止批量无条件回滚状态。

## 方案取舍

- 采用后端定时 worker：无需依赖访问流量，符合现有 worker 架构，应用内可测试和回滚。
- 不采用查询时顺带更新：避免读请求承担配置读取与写库失败。
- 不采用 MySQL Event：避免业务规则分散到数据库运维配置。
